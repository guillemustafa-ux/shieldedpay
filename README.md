# ShieldedPay — private payments on Ethereum: stealth addresses + a compliant privacy pool

> Two layers of on-chain payment privacy, built end-to-end. **Stealth addresses** (ERC-5564 / ERC-6538) hide *who receives*; a **zero-knowledge privacy pool with association sets** (the Privacy Pools design) breaks the deposit↔withdrawal link *while letting an honest user prove their funds are clean*. The withdrawal proof is generated **in the browser** — no secret ever leaves the user's machine.

## Why this exists

On-chain privacy is one of the most heavily backed narratives in crypto right now — Privacy Pools is live on mainnet, the Ethereum Foundation integrated it into its Kohaku wallet, and EIP-8182 proposes a protocol-native shielded pool. The interesting frontier is not "a mixer" (that's a solved, sanctioned dead end) but **privacy that a good actor can prove is compliant**. ShieldedPay implements that mechanism — association sets — and explains it.

The full argument, with the market context and a comparison against Tornado / Railgun / Aztec, is in **[`docs/THESIS.md`](docs/THESIS.md)**. Read that to judge whether the author understands the space. For the engineering view — where this demo's structural boundaries are, what a production version would require, and why the highest-leverage move is to build *on* the ecosystem's shared pool rather than beside it — see **[`docs/ARCHITECTURE.md`](docs/ARCHITECTURE.md)**.

## Architecture

```
FASE A — Stealth addresses (receiving privacy)
  Bob registers a stealth meta-address ──► ERC6538Registry
  Alice derives a fresh stealth address off-chain (ECDH, secp256k1)
  Alice pays it and announces  ──────────► ERC5564Announcer  ──► Bob scans, derives the private key, sweeps

FASE B — Privacy Pool with association sets (unlinkable, compliant transfer)

  deposit(commitment)                         withdraw(proof, root, associationRoot, …)
        │                                             │
        ▼                                             ▼
  ┌───────────────┐   commitment = Poseidon(     ┌──────────────────────────────────┐
  │ PrivacyPool   │      nullifier, secret)      │ ZK proof (generated in the browser):│
  │  Merkle tree  │◄─── inserted as a leaf       │  • commitment ∈ state tree (root)   │
  │  (Poseidon,   │                              │  • commitment ∈ association set      │
  │   history)    │                              │        (associationRoot) ── clean   │
  └───────────────┘                              │  • nullifierHash unspent            │
        ▲                                        │  • recipient/relayer/fee bound      │
        │ associationRoot (clean subset)         └──────────────────────────────────┘
  ┌───────────────┐                                       │ verified on-chain (Groth16)
  │ ASP (owner)   │ publishes roots of the                ▼
  │               │ off-chain "clean" tree        pays recipient (− fee to relayer)
  └───────────────┘                               marks nullifier spent
```

The key property: the withdrawal proves membership in **both** the pool and the clean association set, in zero knowledge, without revealing *which* deposit is being withdrawn. A deposit the ASP never marked clean cannot withdraw.

## The ZK circuit

`withdraw.circom` proves, for a secret `(nullifier, secret)`:

1. `commitment = Poseidon(nullifier, secret)` is a leaf of the pool's state tree (`root`);
2. the **same** commitment is a leaf of the association tree (`associationRoot`) — the Privacy Pools compliance proof;
3. `nullifierHash = Poseidon(nullifier)` is published so the pool can mark it spent (no double-spend);
4. `recipient`, `relayer` and `fee` are bound into the proof, so a relayer can't redirect funds after receiving it.

The trusted setup reuses `powersOfTau28_hez_final_15.ptau` from the public Hermez / Perpetual Powers of Tau ceremony, verified against its published Blake2b hash. No home-rolled setup.

## Client-side proving (the dApp)

The [`frontend/dapp`](frontend/dapp) app (Vite + React + ethers v6) generates the Groth16 withdrawal proof **in the browser** with snarkjs (wasm): it reconstructs the Merkle tree from on-chain `Deposit` events, builds the paths, and proves locally. The `(nullifier, secret)` never leaves the user's device — this is the whole point of client-side proving, and it's rarely shown in portfolio pieces.

## Tests & invariants

```bash
forge test -vv          # 36 tests: unit + fuzz + stateful invariant
cd circuits && npm test  # 4 circuit tests (valid + 3 invalid cases)
```

Highlights:
- **`test_MerkleRoot_MatchesJsHarness`** — the on-chain Poseidon tree produces the *exact same root* as the off-chain JS harness; the integration is bit-for-bit correct.
- **`test_Withdraw_ValidProof_PaysRecipient`** — a **real** Groth16 proof is verified on-chain and pays out.
- **`test_Withdraw_RevertsIf_AssociationRootNotPublished`** — a deposit outside the clean set cannot withdraw (the compliance guarantee).
- Plus double-spend, invalid-proof, unknown-root, and fee-bound reverts, and a stateful invariant (`pool balance == denomination × (deposits − withdrawals)`).

## Security & honest scope

This is a demonstration of a mechanism, not a product. The ASP is a single owner (not decentralized), anonymity depends on set size, and it has not been audited. The trust model, adversarial review, and **known residual risks** are documented candidly in **[`SECURITY.md`](SECURITY.md)** — being precise about the boundary is part of the work.
