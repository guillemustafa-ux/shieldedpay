# Security

> **ShieldedPay is an educational / portfolio project. It has NOT been audited. Do NOT use it with real funds.** This document is an honest account of what was verified, what is trusted, and what remains a known risk — not a security guarantee.

## Verification performed

- **Full Foundry test suite green** (`forge test`): unit + fuzz + stateful invariant tests across both phases (stealth addresses and the privacy pool). The pool suite verifies a **real Groth16 proof on-chain**, not a mock.
- **Off-chain / on-chain consistency:** `test_MerkleRoot_MatchesJsHarness` asserts that the on-chain Poseidon Merkle tree produces the exact same root as the off-chain JS harness (`circuits/test/merkleTree.js`) for the same insertions. If the on-chain hasher or tree logic diverged from the circuit, neither the root check nor the real-proof withdrawal would pass.
- **Circuit checks:** the withdraw circuit is exercised with one valid case and three invalid cases (commitment outside the state tree, commitment outside the association set, wrong nullifier hash), each rejected as expected.
- **Trusted setup provenance:** the Groth16 proving/verifying keys derive from the public **Perpetual Powers of Tau (Hermez)** `.ptau` (`powersOfTau28_hez_final_15.ptau`), downloaded from the official iden3/Polygon mirror and verified against its published Blake2b-512 hash (exact match). No home-rolled trusted setup.

## Trust model

- **Poseidon hasher:** deployed on-chain from bytecode generated by `circomlibjs` (`poseidonContract.createCode(2)`) — the same Poseidon implementation the circuit uses — so on-chain and in-circuit hashing are identical by construction.
- **Groth16 verifier:** generated by `snarkjs` from the circuit; not hand-written. Trust reduces to the trusted setup (above) and the soundness of Groth16 over BN254.
- **Association Set Provider (ASP):** the **deployed** pool uses a single `Ownable` account that publishes association roots, and users trust that owner to include honest deposits and exclude flagged ones. **This is a deliberate simplification for the demo** — the repo also ships a decentralized-ASP implementation (`ASPRegistry` + on-chain fraud proofs) that removes this trust; see Residual risks and [`docs/DECENTRALIZED-ASP.md`](docs/DECENTRALIZED-ASP.md).
- **Relayer:** a withdrawal may be submitted by a third-party relayer for a `fee`. The `recipient`, `relayer` and `fee` are bound into the proof (`recipient`/`relayer`/`fee` are public inputs), so a malicious relayer cannot redirect funds or inflate the fee after receiving the proof — any change invalidates verification.

## Adversarial review

Threats considered and how they are handled:

- **Double-spend / withdrawing the same deposit twice** → each withdrawal consumes a unique `nullifierHash`, marked spent before any transfer (`nullifierHashes[nullifierHash] = true`). Checked by `test_Withdraw_RevertsIf_NullifierAlreadySpent`.
- **Reentrancy on withdraw** → `nonReentrant` guard plus strict checks-effects-interactions: state (nullifier spent) is written before the external `call` that pays the recipient/relayer.
- **Withdrawal front-running / proof theft** → the proof binds `recipient`. Copying a pending withdrawal transaction only pays the original intended recipient, and the nullifier is spent on first inclusion, so there is nothing to steal.
- **Forging membership** → withdrawing a deposit that is not in the pool, or not in the clean association set, requires a valid Merkle path the attacker does not have; the proof does not close. The association-exclusion path is checked by `test_Withdraw_RevertsIf_AssociationRootNotPublished`.
- **Stale / unknown roots** → the pool keeps a bounded circular history of recent state roots and only accepts a `root` within it (`isKnownRoot`); an arbitrary root is rejected (`test_Withdraw_RevertsIf_UnknownStateRoot`).
- **Field-range abuse** → commitments are required to be `< FIELD_SIZE` on deposit; the generated verifier range-checks all public signals, rejecting out-of-field inputs.
- **Fee stranded without a relayer** → a withdrawal with `fee > 0` and `relayer == address(0)` would leave the fee stuck in the contract. The pool rejects it up front (`require(relayer != address(0) || fee == 0)`), so the balance invariant holds universally. Checked by `test_Withdraw_RevertsIf_FeeWithoutRelayer`.

## Static analysis (Slither)

Slither was **not run in the development environment** for this build (the local Windows toolchain did not have Slither installed), so this document does not report Slither output it did not produce. Running it is recommended before any further use:

```bash
pip install slither-analyzer
slither . --filter-paths "lib|test|src/verifiers"
```

The generated verifier (`src/verifiers/WithdrawVerifier.sol`) is expected to raise assembly / low-level warnings that are inherent to snarkjs output and are not defects. Any findings on the hand-written contracts (`PrivacyPool`, `ASP`, `MerkleTreeWithHistory`) should be triaged and recorded here.

## Residual risks (known, accepted)

- **Centralized ASP (in the deployed demo).** The deployed pool's ASP is a single owner: a malicious or compromised owner could refuse to include honest deposits (censorship) or publish a permissive set. The repo addresses this with a **decentralized ASP** — a staked, multi-provider `ASPRegistry` where a lying provider is slashable by an on-chain fraud proof (`challengeIntegrity` / `challengeDegenerate` / `challengeInclusion`), implemented and tested but **not deployed** (see [`docs/DECENTRALIZED-ASP.md`](docs/DECENTRALIZED-ASP.md)). Two limits remain even there: censorship (wrongful *exclusion*) is not slashable and is handled by multi-ASP exit rather than a fraud proof, and taint propagation is not recomputed on-chain (it collapses to an off-chain attester's published flags).
- **Anonymity set size.** Real privacy depends on many deposits, relayer usage, and defenses against timing/gas-metadata correlation. A demo pool with few deposits provides little practical anonymity regardless of the cryptography.
- **Trusted setup assumption.** Groth16 soundness assumes the Powers of Tau ceremony had at least one honest participant. This repo relies on the well-known Hermez ceremony rather than proving that property itself.
- **No upgrade path / no emergency stop.** The contracts are intentionally minimal; there is no pause or upgrade mechanism.
